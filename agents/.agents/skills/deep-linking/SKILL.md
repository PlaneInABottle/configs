# Deep Linking

## Overview

Deep linking enables external URLs to open specific app screens. This skill covers URL schemes, universal links (iOS), app links (Android), Expo Router linking configuration, and OAuth redirect handling. Composes with `react-native-expo` (routing), `supabase` (OAuth redirects), and `maestro-testing` (E2E link verification).

## Quick Start

Three steps to enable deep linking in an Expo project:

**1. Configure scheme in `app.json`:**

```json
{
  "expo": {
    "scheme": "myapp",
    "ios": { "bundleIdentifier": "com.example.myapp" },
    "android": { "package": "com.example.myapp" }
  }
}
```

**2. Set up Expo Router linking config:**

Expo Router auto-generates linking from the file structure. Your `app/_layout.tsx` handles deep link routing automatically — no manual `linking` config needed. See [Expo Router Linking Config](#expo-router-linking-config) below.

**3. Test:**

```bash
# iOS Simulator
xcrun simctl openurl booted "myapp://profile"

# Android Emulator
adb shell am start -a android.intent.action.VIEW -d "myapp://profile"
```

Or use the bundled script: `bash scripts/test-deep-link.sh "myapp://profile"`

## URL Schemes

Custom URL schemes let the app handle `myapp://path` links.

### Setup

The `scheme` field in `app.json` is the single source of truth:

```json
{
  "expo": {
    "scheme": "myapp"
  }
}
```

Expo config plugin generates the native config:
- **iOS:** `CFBundleURLSchemes` in `Info.plist`
- **Android:** `<intent-filter>` with `scheme="myapp"` in `AndroidManifest.xml`

### Multiple Schemes

Pass an array for multiple schemes (rare; prefer one):

```json
{
  "expo": {
    "scheme": ["myapp", "myappstaging"]
  }
}
```

### Testing

```bash
# iOS — open in booted simulator
xcrun simctl openurl booted "myapp://settings?tab=account"

# Android — broadcast intent to running app
adb shell am start -a android.intent.action.VIEW -d "myapp://settings?tab=account"
```

### Limitations

- URL schemes are **not secure** — any app can register the same scheme.
- They do **not** survive app installation (no fallback if app is not installed).
- Use universal/app links for production; URL schemes are best for OAuth redirects and dev testing.

## Universal Links (iOS)

Universal Links open your app when users tap `https://yourdomain.com/path` links, with graceful fallback to Safari if the app is not installed.

### Key Components

1. **AASA file** — JSON hosted at `https://yourdomain.com/.well-known/apple-app-site-association`
2. **Associated domains entitlement** — Declares which domains your app handles
3. **Xcode project** — Generated by Expo; must include the entitlement

See `references/ios-universal-links.md` for the full configuration guide.

### Quick Checklist

- [ ] AASA file hosted at your domain (HTTPS, no redirects)
- [ ] `associatedDomains` configured in `app.json`:
  ```json
  {
    "expo": {
      "ios": {
        "associatedDomains": ["applinks:yourdomain.com", "applinks:*.yourdomain.com"]
      }
    }
  }
  ```
- [ ] Expo builds the Xcode entitlement correctly
- [ ] Test: `xcrun simctl openurl booted "https://yourdomain.com/profile"`

## App Links (Android)

App Links are the Android equivalent of Universal Links — `https://yourdomain.com/path` opens the app directly.

### Key Components

1. **`assetlinks.json`** — Hosted at `https://yourdomain.com/.well-known/assetlinks.json`
2. **Intent filter** — Declared in `AndroidManifest.xml` (Expo handles this)
3. **Digital Asset Links API** — Google's verification system

See `references/android-app-links.md` for the full configuration guide.

### Quick Checklist

- [ ] `assetlinks.json` hosted at your domain (HTTPS)
- [ ] Intent filters configured via `app.json`:
  ```json
  {
    "expo": {
      "android": {
        "intentFilters": [
          {
            "action": "VIEW",
            "data": [{ "scheme": "https", "host": "yourdomain.com", "pathPrefix": "/" }],
            "category": ["BROWSABLE", "DEFAULT"]
          }
        ]
      }
    }
  }
  ```
- [ ] Test: `adb shell am start -a android.intent.action.VIEW -d "https://yourdomain.com/profile"`

## Expo Router Linking Config

Expo Router uses file-based routing — the file system *is* the linking config. No manual `linking` object needed.

### How It Works

```
app/
├── _layout.tsx          # Root layout
├── index.tsx            # Handles /
├── profile.tsx          # Handles /profile
├── settings/
│   └── [id].tsx         # Handles /settings/:id (dynamic)
└── (auth)/
    ├── login.tsx        # Handles /login
    └── callback.tsx     # Handles /auth/callback (OAuth)
```

Each file maps to a route. Deep links resolve naturally:
- `myapp://profile` → `app/profile.tsx`
- `myapp://settings/42` → `app/settings/[id].tsx` (params: `{ id: "42" }`)
- `myapp://login` → `app/(auth)/login.tsx`

### Accessing Deep Link Params

```tsx
// app/settings/[id].tsx
import { useLocalSearchParams } from 'expo-router';

export default function SettingsScreen() {
  const { id } = useLocalSearchParams<{ id: string }>();
  // ...
}
```

### Handling the Initial URL

```tsx
import { usePathname } from 'expo-router';

// Expo Router handles initial URL automatically.
// To read it imperatively (rare):
import * as Linking from 'expo-linking';

const initialUrl = await Linking.getInitialURL(); // "myapp://profile"
```

### Custom Scheme + Host

If you need a custom host (not just scheme), set `scheme` at the top level and `origin` inside the `expo-router` plugin:

```json
{
  "expo": {
    "scheme": "myapp",
    "plugins": [
      [
        "expo-router",
        {
          "origin": "https://yourdomain.com"
        }
      ]
    ]
  }
}
```

## OAuth Redirect Pattern

OAuth flows (Google, Apple Sign-In) require deep linking to return the user to the app after authentication.

### Supabase + Deep Linking Flow

```
1. App → opens `https://yourproject.supabase.co/auth/v1/authorize?provider=google&redirect_to=myapp://auth/callback`
2. Browser → Google OAuth consent
3. Google → redirects to `https://yourproject.supabase.co/auth/v1/callback`
4. Supabase → exchanges code for session → redirects to `myapp://auth/callback#access_token=...&refresh_token=...`
5. App → receives tokens via deep link, sets session
```

### Implementation

```tsx
// app/(auth)/callback.tsx
import { useEffect } from 'react';
import { useLocalSearchParams, useRouter } from 'expo-router';
import { supabase } from '../../lib/supabase';
import * as Linking from 'expo-linking';

export default function AuthCallback() {
  const router = useRouter();

  useEffect(() => {
    // Supabase SDK extracts tokens from the URL fragment automatically
    const handleAuth = async () => {
      const url = await Linking.getInitialURL();
      if (url) {
        // Exchange the callback URL for a session (PKCE flow)
        const { data, error } = await supabase.auth.exchangeCodeForSession(url);
        if (error) {
          console.error('Auth callback error:', error);
          router.replace('/(auth)/login');
        } else if (data.session) {
          router.replace('/(tabs)');
        }
      }
    };
    handleAuth();
  }, []);

  return <LoadingSpinner />;
}
```

### Redirect URL Configuration

In Supabase Dashboard → Authentication → URL Configuration:
- **Site URL:** `myapp://`
- **Redirect URLs:** `myapp://auth/callback`

In `app.json`, ensure the `auth/callback` route exists:
```
app/(auth)/callback.tsx → handles myapp://auth/callback
```

### Expo AuthSession (Alternative)

For non-Supabase OAuth, use `expo-auth-session`:

```tsx
import * as AuthSession from 'expo-auth-session';

const redirectUri = AuthSession.makeRedirectUri({ scheme: 'myapp', path: 'auth/callback' });
// Produces: "myapp://auth/callback"
```

## Anti-Patterns

| Don't | Do Instead |
|-------|-----------|
| Use `Linking.addEventListener` for routing | Use Expo Router file-based routing (auto-handles deep links) |
| Hardcode redirect URLs | Use `AuthSession.makeRedirectUri` or scheme from `app.json` |
| Skip AASA/assetlinks testing | Test with `xcrun simctl openurl` / `adb shell am start` before release |
| Use HTTP for AASA/assetlinks | HTTPS is required — no exceptions |
| Register multiple schemes for prod | Use one scheme + universal/app links; schemes are for dev/OAuth |
| Parse OAuth tokens manually from URL | Let Supabase SDK handle `getSession()` from the callback URL |
